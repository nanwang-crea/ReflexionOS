# ReflexionOS 记忆系统第一期：最小连续性闭环方案

> 状态：当前阶段方案文档。
> 该文档定义记忆系统第一期的产品切片与交付边界。
> 这是一份服务于总体蓝图的阶段方案，不展开实现任务树。

**版本**: v1.0  
**日期**: 2026-04-20  
**对应上位文档**: `docs/superpowers/specs/2026-04-19-memory-architecture-v1.md`

---

## 一、定位

第一期不是在实现完整记忆系统，而是在从总体蓝图中抽取一个可闭环、可验证、可感知价值的最小子系统。

这期要做的，不是：

- 完整的长期记忆平台
- 跨会话连续性系统
- 智能记忆治理体系

而是：

## 当前项目 + 当前会话 的最小连续性闭环

一句话定义：

**让 agent 在同一个项目里持续推进任务时，不会“失忆式重启”。**

---

## 二、第一期唯一目标

第一期的唯一目标不是“记住更多”，而是：

## 让连续协作不断片

它具体体现为三件事：

### 2.1 Thread continuity

用户说：

- 刚刚我们在聊什么
- 继续刚才的话题

系统能接上当前线程。

### 2.2 Task continuity

用户说：

- 继续上次那个问题
- 按刚才说的方向继续推进

系统不会重复从头分析。

### 2.3 Commitment continuity

用户说：

- 你刚才答应我的那个事
- 那个待办继续做

系统不会把承诺和 follow-up 丢掉。

---

## 三、第一期范围：从总体架构中抽出的最小子集

第一期只抽取总体架构中的以下部分：

### 3.1 `Active Workspace`

这是第一期的核心中的核心。

它是当前项目、当前会话的工作记忆层。

第一期只保留四类核心内容：

- `Thread`
  - 当前在聊什么
- `Task State`
  - 当前在做什么
- `Commitment`
  - 当前答应了什么、还欠着什么
- `Recent Reflection`
  - 最近尝试后的经验反馈

### 3.2 `Archive`

第一期只做最小归档能力：

- session log
- execution log
- tool result log

第一期的 Archive 目标只有一个：

**不丢。**

它不是第一期的智能能力中心。

### 3.3 `Candidate`

第一期只做极简候选区：

- 只支持“用户明确要求记住”的内容进入候选

例如：

- 记住这个 bug root cause
- 记住这个 API 设计约束

第一期不做：

- 自动候选生成
- 模式挖掘
- 自动晋升

### 3.4 `Durable Memory`

第一期不做完整的长期记忆体系。

只保留最小项目级长期记忆能力：

- 规则
- 高价值上下文
- 决策记录

它不是第一期的主要价值来源。

第一期的主要价值来自 `Active Workspace`。

---

## 四、第一期不做什么

为了保证产品切片足够小，第一期明确不做以下内容：

### 4.1 不做跨会话长期记忆主能力

跨会话连续性保留在总体蓝图里，但不作为第一期主交付目标。

### 4.2 不做自动晋升机制

不做：

- Candidate -> Durable 自动流转
- 自动模式提炼
- 自动长期沉淀

### 4.3 不做复杂记忆编译器

第一期不做：

- 多源推理
- 多策略 recall
- ranking system
- 高复杂度合并逻辑

### 4.4 不做 Archive 智能检索能力

第一期的 Archive 只负责记录，不负责变聪明。

不做：

- 向量检索
- embedding recall
- 复杂全文索引优化
- graph memory
- knowledge graph

### 4.5 不做 dreaming / consolidation

第一期不做：

- consolidation
- dreaming
- 自动长期经验沉淀

---

## 五、第一期的运行时形态

第一期不实现完整的运行时记忆体系，而是启用其中的最小子集。

### 5.1 第一版 Memory Compiler

第一期不做完整 `Memory Compiler`，只做一个降级版：

## `State Assembler`

它只做一件事：

**把当前 Active Workspace 整理成稳定可用的工作视图。**

它不负责：

- 多层记忆调度
- 复杂检索
- 多源冲突裁决
- 智能重排

### 5.2 第一版 Working View

第一期的工作视图建议保留五块，不能再继续压缩：

1. `当前状态`
   - 现在在聊什么、现在在做什么
2. `当前任务`
   - 当前目标、最近进度、待继续事项
3. `当前承诺`
   - 当前待办、follow-up、答应用户的事
4. `最近反思`
   - 最近失败点、有效策略、注意事项
5. `下一步`
   - 当前最合理的下一动作

注意：

第一期不建议把工作视图压缩成只有：

- `STATE`
- `TASK`
- `NEXT`

因为这样会把：

- Commitment
- Reflection

折叠掉，而这两者正是第一期连续性体验的关键。

### 5.3 第一版运行时读取

第一期运行时只启用蓝图中的最小读取子集：

- 固定读取 `Active Workspace`
- 极弱使用项目级长期记忆
- `Archive` 只做必要时回查

更准确地说：

第一期不是“不启用运行时模型”，而是：

## 只启用运行时模型中的最小可用子集

---

## 六、第一期系统闭环

第一期最终应形成如下最小闭环：

```text
用户输入
  ↓
当前工作区更新
  ↓
状态整理器（State Assembler）
  ↓
工作视图（当前状态 / 当前任务 / 当前承诺 / 最近反思 / 下一步）
  ↓
LLM 执行
  ↓
更新当前工作区 + 写入归档
```

这就是第一期真正要跑通的闭环。

---

## 七、第一期的真实产品价值

第一期的价值不在于：

- 记住了多少条 memory
- 存了多少 archive
- 建了多少候选项

而在于：

### 7.1 少重复解释

用户不需要反复重新讲当前背景和当前方向。

### 7.2 少重复分析

agent 不会每次都从零开始看问题。

### 7.3 少重复承诺

系统不会丢掉刚刚说过要做的事。

---

## 八、第一期的成功标准

第一期成功，不靠“功能列表完成”，而靠连续性指标。

建议至少用这三个指标衡量：

### 8.1 Thread continuity success

“刚刚我们在聊什么”“继续刚才的话题”是否能稳定接上。

### 8.2 Task continuity success

“继续上次那个问题”是否不会退化成重新从头分析。

### 8.3 Commitment continuity success

“你刚才答应我的那个事”是否还能被恢复出来。

如果要压成一句话：

## 第一阶段最重要的指标，不是记住了多少，而是用户少重复解释了多少次。

---

## 九、第一期与总体蓝图的关系

第一期不是完整实现总体蓝图，而是从蓝图里切出一个最小连续性内核。

对应关系如下：

| 总体蓝图层 | 第一期状态 |
| --- | --- |
| Archive | 仅做写入与最小回查 |
| Active Workspace | 第一优先，核心交付 |
| Candidate | 极简版，仅支持用户显式记忆 |
| Durable Memory | 只保留最小项目级能力 |
| Memory Compiler | 降级为 State Assembler |
| Working View | 做五块最小工作视图 |
| 跨会话连续性 | 暂不作为主目标 |

所以第一期不是在“实现完整架构”，而是在：

## 实现总体架构中的最小连续性闭环

---

## 十、最终判断

第一期如果做得过大，会变成：

- 复杂但不稳定的 memory system

第一期如果按本方案切，会更像：

- 稳定可用的 continuity engine v0

因此，第一期最正确的开工方式不是追求“完整记忆系统”，而是：

## 先把当前项目、当前会话中的最小连续性闭环做出来

这也是当前总体蓝图最自然的第一落点。
