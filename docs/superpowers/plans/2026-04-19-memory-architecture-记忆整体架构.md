# ReflexionOS 记忆架构 V1 方案

> 状态：当前聚焦设计文档。
> 该文档描述 ReflexionOS 最终要建设成的记忆系统形态。
> 这是一份产品与架构文档，不是实现清单。

**版本**: v1.1  
**日期**: 2026-04-20  
**状态**: 当前记忆方向主文档

---

## 一、背景与问题定义

ReflexionOS 当前已经具备本地执行型代码代理的核心闭环：

- 项目管理
- 会话工作区
- 执行记录
- 工具回执
- 本地持久化

但要从“会做事的工具”走向“能长期协作的助手”，还缺少一套真正的记忆系统。当前缺口主要体现在：

- 用户说“继续刚才那个 bug”时，agent 容易重新开始
- 用户说“刚刚我们在聊什么”或“继续刚才的话题”时，系统缺少稳定的线程状态
- 用户说“你刚才答应我的那个事”时，系统缺少承诺与责任状态
- 项目规则、稳定上下文和高成本获取的信息无法持续生效
- 每次执行的经验难以被后续任务和后续会话有效复用
- 原始记录虽然存在，但缺少系统化的归档、晋升、召回与追溯机制

因此，ReflexionOS 需要的不是一个“大而全的智能记忆平台”，而是一套面向长期协作的透明记忆系统。

---

## 二、总体架构目标

ReflexionOS 记忆系统的总体目标是：

**构建一个面向长期协作的透明记忆系统，使 agent 能持续承担上下文、责任与推进方向，在当前会话、当前项目以及跨会话协作中都保持连续性，同时让少量高价值知识稳定影响未来。**

这套系统最终应支持：

- 自然接话
- 持续推进
- 承诺不丢
- 项目知识持续生效
- 高成本上下文不被浪费
- 所有长期记忆都可检查、可编辑、可删除、可追溯

如果压缩成一句话：

**ReflexionOS 的记忆系统，不是为了记住所有历史，而是为了让 agent 持续承担上下文、责任与推进方向。**

## 三、这套记忆系统最终要解决什么问题

### 3.1 当前会话连续性

用户在同一个项目、同一个会话中持续推进任务时，系统要能回答：

- 刚刚我们在聊什么
- 当前在推进哪条线
- 已经形成了什么结论
- 还有什么没做
- 你刚才答应了什么

### 3.2 当前项目连续性

即使用户切换话题或任务，系统也要能保持项目级稳定认知，例如：

- 这个项目的技术栈
- 关键目录和常用命令
- 已确认规则
- 稳定上下文
- 决策记录

### 3.3 跨会话连续性

当用户在同一项目下重新打开会话时，系统要能逐步支持：

- 恢复前一条线程
- 继续上一轮未完成的任务
- 恢复被保留的承诺与 follow-up
- 复用先前已确认的规则、上下文和决策

### 3.4 长期协作可信度

系统不仅要“接得上”，还要“接得对”。这要求：

- 长期记忆不能是黑箱
- 每条长期记忆必须能追溯证据
- 记忆项必须有生命周期
- 记忆项必须能被验证、修正和删除

## 三点五、当前已实现的基础层与未来记忆层的边界

当前仓库已经完成的，不是完整记忆系统，而是记忆系统的基础层。

更准确地说，当前已实现的是：

- 后端正式 `session` 资源
- 后端 `session history` 的 `round -> items` 真相源
- execution 的 `completed / cancelled / failed` 边界语义
- 前端 session summary 缓存与 history 按需加载
- 前端 draft round / overlay 的瞬时执行态

这意味着当前已经具备：

- 会话连续性的正式资源基础
- transcript 级归档与回放基础
- 后端作为 session/history 真相源的边界

但当前**尚未**实现：

- 结构化 `Active Workspace` 数据模型
- `Candidate` 候选区
- `Durable Memory` 长期记忆区
- 可治理的 memory promotion / review / recall 机制
- 面向记忆层的统一 `State Assembler`

因此，后文的四区模型与长期记忆机制，应被理解为：

- 当前系统的目标架构方向
- 而不是“已经落地的现状”

## 四、核心设计哲学

### 4.1 Archive 不等于 Memory

这是整个系统最重要的原则。

- `Archive` 的目标是“不丢”
- `Memory` 的目标是“持续影响未来”

所有原始记录都可以留在 Archive 中，但只有少量高价值信息才应成为真正的 Memory。

### 4.2 协作连续性优先

记忆系统首先服务于：

- 当前会话不断片
- 当前任务不断片
- 当前承诺不断片

而不是先服务于“大规模长期知识沉淀”。

### 4.3 少写入，优召回

系统不追求“记得更多”，而追求：

- 记得少
- 记得准
- 召回稳定
- 影响明确

### 4.4 长期记忆必须显式、可检查、可编辑

ReflexionOS 的产品定位强调透明与可控，因此长期记忆必须满足：

- 可见
- 可编辑
- 可删除
- 可审核
- 可回查来源

### 4.5 自动归档，谨慎晋升

系统必须默认：

- Archive 自动写入
- 长期记忆谨慎晋升

第一阶段采用人工门控晋升；后续即使引入更智能的候选机制，也不应轻易取消长期记忆的审核逻辑。

## 五、最终系统的连续性层次

这套记忆系统最终要覆盖三层连续性：

### 5.1 当前会话连续性

这是最靠近用户体验的一层，负责：

- 线程状态
- 任务状态
- 承诺状态
- 最近反思

### 5.2 当前项目连续性

这是项目级稳定认知层，负责：

- 项目规则
- 高价值上下文
- 决策记录
- 可复用的项目知识

### 5.3 跨会话连续性

这是更长期的协作层，负责：

- 同项目不同会话之间的线程恢复
- 历史任务接续
- 长期承诺与 follow-up 的恢复
- 长期记忆的稳定复用

## 六、总体架构：四区模型

ReflexionOS 记忆系统最终采用四区模型：

这里的“四区模型”描述的是目标态架构，不代表当前代码已经完整具备四区实现。

当前真实已落地的部分，主要集中在：

- `Archive` 的最小 transcript/session history 基础
- 与 `Active Workspace` 相关的极弱雏形（执行上下文、session 选择、history 恢复）

而 `Candidate` 与 `Durable Memory` 当前仍停留在设计层。

1. `Archive 区（归档区）`
2. `Active Workspace 区（当前工作区）`
3. `Candidate 区（候选区）`
4. `Durable Memory 区（长期记忆区）`

### 6.1 Archive 区

`Archive` 区保存全部原始记录，是系统的档案层与证据层。

典型内容包括：

- 全部 session transcript
- 全部 execution
- 全部 tool receipts
- 全部 reflection 原始记录
- 项目扫描结果
- memory 变更历史

职责：

- 保留原始事实
- 支持全文搜索、时间回查与证据回查
- 为长期记忆提供来源

Archive 区可以很大，但不应默认全量注入到工作上下文中。

### 6.2 Active Workspace 区

`Active Workspace` 区承载当前项目、当前会话中的活跃协作状态。

典型内容包括：

- 当前线程状态
- 当前任务状态
- 当前承诺与待办
- 最近已完成事项
- 最近待继续事项
- 最近触达文件
- 最近失败点
- 最近若干条 reflection

职责：

- 支撑当前会话中的自然接话
- 支撑当前任务的持续推进
- 支撑当前承诺的持续可见
- 在当前协作中快速注入上下文

这是系统最贴近实时交互的一层。

### 6.3 Candidate 区

`Candidate` 区是长期记忆的候选缓冲区。

它不是 Archive，也不是 Durable Memory，而是一个审核与晋升通道。

典型来源包括：

- 用户明确要求记住的内容
- 多次重复出现的 correction
- 已被确认的项目决策
- 高价值上下文候选
- 被验证过的模式性经验

职责：

- 拦住低质量信息
- 提供审核入口
- 为晋升过程保留来源与证据

### 6.4 Durable Memory 区

`Durable Memory` 区是少量长期记忆的承载区。

典型内容包括：

- 项目规则
- 已确认的高价值上下文
- 决策记录
- 经验证值得跨会话保留的少量线程/承诺信息

职责：

- 在长期范围内稳定影响未来任务
- 为 agent 提供项目级持续约束
- 支持跨会话连续性
- 在需要时被召回，而不是每次全量注入

## 七、最终对象模型

这套系统在治理/存储层保留细粒度对象，以支持：

- 记忆治理
- 审核与晋升
- 生命周期管理
- 证据追溯
- 用户可理解与可编辑

因此，治理/存储层继续围绕七类对象组织：

### 7.1 Rule

规则类记忆，用于表达持续有效的行为约束。

示例：

- 任何 schema 变更必须带 migration
- 修改接口前必须先读 OpenAPI
- 不要直接大改 UI

### 7.2 Context

上下文类记忆，用于表达高成本获取、值得复用的稳定事实。

示例：

- payment service 的真实 owner 是 infra team
- 某目录不是生产入口
- 某份文档已经过时

### 7.3 Decision

决策类记忆，用于记录已经确认的项目取舍与历史决定。

示例：

- Kafka 替代 Redis Streams
- 当前阶段统一采用 patch-first 修改策略

### 7.4 Thread

线程类记忆，用于表达对话或协作线程状态。

示例：

- 我们当前在讨论记忆架构的总体目标
- 当前形成的结论是“当前会话连续性优先”
- 当前尚未展开的问题是第一阶段保留哪些一级对象

### 7.5 Task State

任务状态类记忆，用于表达当前正在推进的任务状态。

示例：

- 当前目标是什么
- 已完成什么
- 待继续什么
- 最近涉及哪些文件

### 7.6 Commitment

承诺类记忆，用于表达已经形成的待办、follow-up 与责任项。

示例：

- 后面需要补一版当前会话连续记忆方案
- 下一步要把总体目标和第一阶段文档同步更新
- 某项工作由 agent 在当前会话中继续承担

### 7.7 Reflection

复盘类记忆，用于表达一次执行或一次推进后的经验反馈。

示例：

- 上次失败原因是本地 postgres 没启动
- 先跑 build 再 patch 的成功率更高
- 当前最合理的下一步是什么

### 7.8 运行时对象族（3+1）

虽然治理/存储层保留七类对象，但运行时不应直接按七类对象工作。

为了避免：

- 召回路径过复杂
- 提示词污染
- 运行时边界模糊
- “这条到底该存哪、该读哪” 的持续争论

运行时对象应收缩为 `3 + 1` 的对象族：

#### A. State（状态族）

由以下对象编译而成：

- Thread
- Task State
- Commitment

它回答的是：

- 我们现在在干嘛
- 当前推进到哪
- 当前还欠着什么

#### B. Knowledge（知识族）

由以下对象编译而成：

- Rule
- Context
- Decision

它回答的是：

- 这个项目长期应该怎么做
- 哪些稳定事实和约束必须被尊重

#### C. Reflection（反思族）

Reflection 在运行时继续独立保留。

它回答的是：

- 最近尝试学到了什么
- 最近失败和有效策略是什么

它与事实层不同，应继续作为经验层独立存在。

#### D. Archive（归档层）

Archive 在运行时仍保持独立。

它回答的是：

- 原始证据在哪里
- 旧记录如何追溯

一句话概括：

**治理层保留细对象，运行层收敛为对象族。**

## 八、运行时读取模型

前面的四区模型与对象模型，描述的是产品层与治理层的完整结构。

但在真正运行时，模型不应该直接面对这套完整结构。

ReflexionOS 的运行时原则应当是：

**存储与治理可以复杂，但运行时读取必须极简。**

换句话说：

- 用户需要理解完整的记忆系统
- 系统需要维护完整的治理结构
- 但模型运行时只应读取最少必要信息

### 8.1 产品/治理层 与 运行层必须分离

产品与治理层保留完整结构：

- `Archive（归档）`
- `Active Workspace（当前工作区）`
- `Candidate（候选区）`
- `Durable Memory（长期记忆）`
- 全部存储对象类型

运行时读取层只保留三种读取模式：

1. `当前上下文`
2. `相关长期记忆`
3. `归档回查`

其中：

- `Candidate` 不属于运行时读取层
- `Candidate` 只属于治理层
- `Candidate` 用于审核、晋升、验证、去重，而不是用于直接推理

### 8.2 记忆编译器（Memory Compiler）

在治理层与运行层之间，系统需要一个核心组件：

它的职责不是简单检索，而是把原始记忆编译成运行时可消费的工作视图。

真正的主链路应当是：

`原始记忆 -> 工作视图 -> 运行时上下文`

而不是：

`原始记忆 -> 直接塞给模型`

Memory Compiler 的核心职责包括：

#### A. 选择

从以下来源中选择相关内容：

- Active Workspace
- Durable Memory
- 必要时的 Archive

#### B. 合并

把多个来源合并成统一的运行时视图，而不是让模型自己拼装。

#### C. 去冲突

过滤明显重复、过时或不该进入当前上下文的内容。

#### D. 压缩

把治理层对象编译成更短、更稳定的运行时表达。

#### E. 保留来源锚点

即使输出被压缩，系统也应保留必要的来源线索，以支持后续回查。

因此：

**记忆编译器不是检索器，而是治理层与运行层之间的桥。**

### 8.3 两层注入 + 一层回查

#### A. 当前上下文

这是运行时固定注入层。

它回答的是：

**我们现在在干嘛？**

默认只包含：

- 当前线程摘要
- 当前任务状态
- 当前承诺
- 最近反思
- 最近触达文件
- 当前目标

这一层必须：

- 稳定
- 小
- 高频可用
- 服务当前会话连续性

#### B. 相关长期记忆

这是运行时按需召回层。

它回答的是：

**这个项目长期应该怎么做？**

默认只召回少量、可信、与当前问题明确相关的长期记忆，例如：

- 少量规则
- 少量上下文
- 必要的决策
- 极少量仍然有效的长期承诺

这一层必须：

- 少
- 稳
- 可信
- 明确相关

#### C. 归档回查

这是运行时的证据回查模式，而不是默认注入层。

它回答的是：

**证据在哪里？**

只在以下情况下使用：

- 用户追问“你怎么知道”
- 需要回看原始 session / execution / receipt
- 需要做原始记录验证

归档回查服务的是：

- 追溯
- 审核
- 引用

而不是主推理过程。

### 8.4 存储对象不等于工作视图

存储层的对象模型可以比较完整，例如：

- Rule
- Context
- Decision
- Thread
- Task State
- Commitment
- Reflection

但模型最终不应直接消费这套原始存储对象。

运行时需要先把它们编译成更适合模型理解的工作视图。

### 8.5 工作视图协议

模型最终看到的，不应是原始存储对象，而应是 Memory Compiler 输出的标准工作视图。

标准工作视图建议固定为以下几块：

1. `当前状态`
   - 我们现在在做什么
2. `待处理承诺`
   - 当前还挂着哪些承诺与待办
3. `近期反思`
   - 最近一次或最近几次尝试的有效经验
4. `重要知识`
   - 当前问题必须遵守的规则、稳定上下文与直接相关的历史决策
5. `当前推进方向`
   - 当前最值得继续推进的方向，以及系统认为下一步应优先做什么

这意味着：

- Rule / Context / Decision 不必逐条原样注入
- Thread / Task State / Commitment 也不必逐条原样注入
- 模型消费的是工作视图，不是存储清单

### 8.6 读取协议

运行时默认读取协议应明确为：

1. 先读 `当前上下文`
2. 判断是否需要 `相关长期记忆`
3. 只有在必要时才进入 `归档回查`
4. 由 Memory Compiler 输出最终工作视图

而不是：

- 所有层都搜
- 所有对象都读
- 全量历史一起塞进上下文

### 8.7 召回选择器

运行时不应该把整套记忆空间直接交给主模型管理。

系统应先完成选择，再把结果交给模型。

因此，ReflexionOS 最终应有一个系统级召回选择器，用于：

- 判断当前问题类型
- 选择最相关的当前上下文
- 决定是否触发长期记忆召回
- 决定是否需要归档回查
- 过滤不该进入当前上下文的记忆项

这意味着：

**不让模型直接管理记忆，而是让系统先管理记忆，再把精选结果交给模型。**

## 九、写入、晋升与权限原则

### 9.1 写入层次

最终系统会有三类写入：

- `Archive 写入`
  - 默认自动
- `Active Workspace 写入`
  - 默认自动维护
- `Durable Memory 写入`
  - 严格受控

### 9.2 晋升权限是系统核心风险点

Archive 到 Durable Memory 的这一步，决定系统最终是：

- 智能协作系统

还是：

- 高级垃圾堆

因此，长期记忆的晋升必须被单独设计，而不能作为普通写入行为看待。

### 9.3 长期记忆的默认原则

最终系统仍应坚持：

- 原始记录自动归档
- 长期记忆默认谨慎晋升
- 持续影响未来的内容必须可审查

第一阶段使用人工门控；
后续即使引入自动候选，也不应直接取消审核逻辑。

### 9.4 典型可晋升来源

可晋升来源最终包括但不限于：

- 用户明确要求记住
- 多次重复 correction
- 已确认决策
- 被验证过的高价值上下文
- 经多次成功复用的稳定模式

### 9.5 明确禁止的晋升方式

以下内容不应直接进入长期记忆：

- 单次推断
- 单次执行后的未验证猜测
- 没有来源的总结
- agent 自己主观判断“这个很重要”
- 没有责任人、没有状态、没有证据的抽象经验

一句话概括：

**单次猜测不能成为长期记忆。**

### 9.6 记忆所有权模型

ReflexionOS 当前按单实际用户、本地优先模式设计。

在这个前提下：

- 用户是记忆系统的最终拥有者
- AI 是记忆的维护者、提议者、使用者
- AI 不是长期记忆的最终裁决者

这意味着：

#### 用户可以

- 创建记忆
- 修改记忆
- 覆盖旧记忆
- 标记记忆失效
- 批准候选记忆进入长期记忆
- 否决 AI 的记忆提议

#### AI 可以

- 自动写入 Archive
- 自动维护 Active Workspace
- 提出 Candidate
- 建议更新、合并、废弃某些长期记忆
- 在运行时读取已批准的记忆

#### AI 不应默认执行

- 单方面创建长期记忆
- 单方面覆盖已有长期记忆
- 单方面认定冲突已经解决
- 在没有用户确认的情况下用新推断替换旧事实

一句话概括：

**AI 可以管理记忆，用户可以裁决记忆；一旦冲突，用户优先。**

### 9.7 冲突解决原则

长期记忆不是只增不减的集合，而是可演化、可替换、可失效的系统状态。

因此，当新信息与已有长期记忆冲突时，系统不应直接按“最新信息”自动覆盖，而应进入标准冲突处理流程。

默认流程应包括：

#### 第一步：发现冲突

系统识别：

- 新信息与已有长期记忆之间存在事实冲突
- 新信息与已有规则或决策之间存在不一致

#### 第二步：回查历史证据

系统优先回查：

- 原始 session
- execution
- tool receipt
- 历史文档
- 决策记录

也就是说：

**先查证据，不先做判断。**

#### 第三步：请求用户确认

当前阶段不做自动冲突裁决。

系统应将冲突项与相关证据展示给用户，请用户确认：

- 旧记忆仍然有效
- 新信息更准确
- 两者都保留但作用域不同

#### 第四步：按用户最新确认执行

如果用户确认新信息为准，则应以用户最新确认结果为准。

旧记忆可以：

- 被覆盖
- 被标记为 `superseded`
- 被标记为 `invalidated`
- 被标记为 `stale`

系统应优先通过状态迁移解决冲突，而不是直接抹掉历史。

一句话概括：

**在记忆冲突中，先查历史证据，再由用户裁决；用户确认后，以用户最新意见为准。**

## 十、召回架构与召回路径

记忆系统不能采用“把所有东西都搜一遍再塞进 prompt”的方式。

最终系统的召回路径应包括：

### 10.1 当前工作区固定注入

每次响应或执行前，固定注入 `Active Workspace` 中与当前会话最相关的内容，例如：

- 当前线程摘要
- 当前目标
- 当前承诺
- 最近已完成事项
- 最近待继续事项
- 最近触达文件
- 最近若干条 reflection

这是当前会话连续性的主通路。

### 10.2 项目级长期记忆按需召回

当任务与项目相关时，按相关性召回少量 Durable Memory。

只召回少量高价值内容，例如：

- 若干条 rule
- 若干条 context
- 必要时的一条或少量 decision

### 10.3 跨会话恢复

当用户开启同项目下的新会话时，系统应逐步支持恢复：

- 最近相关线程
- 未完成承诺
- 仍然有效的任务状态线索
- 必要的项目级稳定记忆

### 10.4 归档回查

归档回查不是默认 prompt 注入路径，而是证据和历史回查路径。

它主要用于：

- 用户追问“你怎么知道”
- 需要翻回原始 session/execution
- 需要查看旧记录全文

## 十一、证据模型与记忆项元数据

所有会持续影响未来行为的记忆项，都应具备最小元数据。

至少包括：

- `id`
- `type`
- `scope`
- `content`
- `owner`
- `evidence`
- `confidence`
- `status`
- `last_verified`
- `ttl`

其中最关键的是：

### 11.1 owner

表示这条记忆由谁确认、由谁负责或由谁维护。

### 11.2 evidence

必须能够指向记忆来源，例如：

- 某个 session
- 某次 execution
- 某条 tool receipt
- 某次用户确认

### 11.3 status

表示当前状态，例如：

- active
- pending
- completed
- stale
- superseded

### 11.4 last_verified 与 ttl

用于表达：

- 最近一次验证时间
- 默认有效窗口
- 是否需要重新确认

没有这些信息的长期记忆，可信度是不完整的。

## 十二、时间分层与过期原则

### 12.1 时间不决定是否存在，只决定默认表示形式

系统不建议按时间硬删除原始记录，而建议按时间调整：

- 默认呈现形式
- 召回优先级
- 验证状态

### 12.2 近期信息

最近 7 到 30 天的信息默认保持细粒度，优先服务当前会话连续性和近期项目接续。

### 12.3 中期信息

1 到 12 个月的记忆默认以摘要或索引形式表示，必要时再回查原始记录。

### 12.4 久远信息

一年以上的信息默认降权处理，只保留：

- 仍然有效的规则
- 关键决策
- 长期上下文
- 必要的长期承诺或责任记录

其余内容仍在 Archive 中保留，但不默认进入工作记忆。

### 12.5 过期不是删除，而是降权与待验证

对可能变化的记忆项，例如：

- owner
- 目录入口
- 部署路径
- 线程状态
- 未完成承诺

应采用：

- 降权
- 待验证
- 重新确认

而不是默认永久真理化。

## 十三、第一阶段与终局的关系

这份文档描述的是最终要做成的系统形态，而不是第一阶段本身。

第一阶段只是整个系统的第一落点。

### 13.1 第一阶段优先完成什么

第一阶段优先完成：

- 当前会话连续记忆
- 当前线程、当前任务、当前承诺、最近反思
- 基础归档与基础回查
- 最小项目级长期记忆能力

### 13.2 后续阶段逐步补什么

后续阶段再逐步补齐：

- 更成熟的项目级稳定记忆
- 跨会话连续性
- 更复杂的候选评估与晋升机制
- 更成熟的召回排序
- consolidation / dreaming
- 向量检索、知识图谱等增强能力

也就是说：

**先从当前会话连续性切入，但最终目标是完整的长期协作记忆系统。**

---

## 十四、非目标

这套系统最终也明确不以以下方向为核心：

- 把所有聊天记录直接当长期记忆
- 自动人物画像系统
- 黑箱式长期记忆生成
- 纯向量库即记忆
- 什么都 embedding、什么都 semantic recall
- 没有证据链的长期记忆自动生成

---

## 十五、对 ReflexionOS 的最终定义

ReflexionOS 记忆架构 V1 可以被定义为：

**一个以协作连续性为中心、四区式、证据可追溯、长期记忆谨慎晋升的透明记忆系统。**

它的基本法则是：

- 全量归档
- 当前工作区持续维护
- 少量候选
- 谨慎晋升
- 按需召回
- 始终可追溯

这不是一个“记住一切”的系统，而是一个“在不丢失历史的前提下，让上下文、责任与推进方向持续存在”的系统。
