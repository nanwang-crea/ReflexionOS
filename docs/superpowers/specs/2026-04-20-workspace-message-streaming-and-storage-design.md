# ReflexionOS Workspace 长消息流式渲染与消息存储收敛方案

> 状态：当前落地设计文档。
> 该文档用于解决新 UI 下“长消息流式输出导致 workspace 卡顿”的问题，并明确前端与后端在消息历史上的职责边界。
> 这份文档聚焦于消息渲染与存储方案，不展开完整记忆系统实现。

**版本**: v1.0  
**日期**: 2026-04-20  
**语言**: 中文  
**关联文档**: `docs/superpowers/specs/2026-04-20-memory-phase-1-grounding-spec.md`

---

## 一、目标

这次要解决的核心问题不是“流式渲染要不要保留”，而是：

**在保留 token 边流边渲染体验的前提下，修复长消息场景下的 UI 卡顿，并把前端消息存储收敛到可持续的边界内。**

本次方案的目标有四个，其中前 4 项当前已完成最小落地：

1. 修复长消息流式输出时的明显卡顿
2. 保留普通消息下的实时流式体验
3. 让前端不再长期持久化完整消息历史
4. 在新 UI 下收敛成单一消息链路，不保留旧方案并存

---

## 二、问题定义

当前长消息卡顿不是单点问题，而是三类成本叠加：

1. 渲染成本
   - `WorkspaceTranscript` 对当前会话做全量 `items.map(...)` 渲染
   - 长消息通过 `MarkdownRenderer` 做 markdown 解析
   - 流式输出时会反复触发同一条长消息重渲染

2. 状态更新成本
   - `useExecutionOverlay` 当前按 token 频率追加内容
   - 高频状态写入会不断驱动 React 更新

3. 持久化成本
   - 旧方案曾把完整 `sessions.items` 连同会话一起持久化到 localStorage
   - 当前已收敛为前端轻量缓存 + 后端 history 真相源，但长 transcript 的渲染成本仍然存在

因此，当前问题不能被理解为“某个组件写得慢”，而应被理解为：

**旧链路曾把高频流式更新、长 markdown 渲染和大对象本地持久化耦合在一起；当前方案要避免重新回到这条路径。**

---

## 三、设计原则

### 3.1 保留流式体验，不做整体降级

普通消息仍然保持 token 边流边渲染。

这次修复不是把流式输出改成“一次性整段出现”，而是只在长消息场景下降低刷新频率。

### 3.2 稳定优先于极限实时

在长消息场景下，优先保证：

- UI 不抖
- 不明显掉帧
- 输入和滚动不被明显阻塞

这意味着在长消息模式下，允许视觉上的流式刷新频率略低于 token 到达频率。

### 3.3 前端缓存，后端真相源

前端负责展示和短期缓存，后端负责完整历史与真相源。

前端不再承担完整历史消息的长期持久化职责。

### 3.4 新 UI 只保留一条链路

在新 UI 下，不保留“旧的前端全量消息持久化方案”和“新的后端拉取方案”双轨并存。

新的消息链路建立后，旧路径应直接清除。

---

## 四、目标架构

### 4.1 前端职责（当前已落地）

前端负责：

- `workspaceStore` 中的 UI 状态
- `sessionStore` 中的 session summary 缓存与已加载 history
- 当前可见会话内容的渲染
- 最近 `10` 轮消息缓存
- 正在流式输出的 overlay 内存态
- 未来的轻量 Active Workspace snapshot 缓存（尚未落地）

前端不负责：

- 完整消息历史真相源
- 完整 execution / receipt 原始归档
- 完整承诺历史与反思历史

### 4.2 后端职责（当前已落地的最小版本）

后端负责：

- session 级 transcript archive 存储
- execution / tool receipt / archive 真相源
- 按 session 拉取 transcript history
- 后续可注入运行时的结构化工作视图

### 4.3 新消息链路（当前实现）

当前链路已收敛为：

```text
用户发消息
  ↓
前端创建当前轮的流式 overlay
  ↓
WebSocket 持续接收 token / receipt / 状态
  ↓
前端按稳定刷新策略更新当前轮 UI
  ↓
当前轮结束后，前端清理 draft 并刷新后端 session history
  ↓
前端继续以轻量缓存方式保留当前 UI 所需数据；更早历史由后端 session history 提供
```

---

## 五、最近 10 轮的定义

本方案中的“一轮”定义为：

- 1 条用户消息
- 对应的 1 次 assistant 连续回复过程
- 该次回复过程中产生的 receipts / status / agent update 一并归属于这一轮

因此，一轮不是单个 `WorkspaceChatItem`，而是一次完整请求及其结果集合。

采用按“轮”缓存而不是按“item”缓存，原因有三个：

1. 更符合用户理解的会话语义
2. 裁剪时不会把一轮内容截断成不完整碎片
3. 更适合后续与 thread/task/commitment 等结构化状态对接

前端长期缓存当前只保留最近 `10` 轮。

更早的历史不是删除，而是由后端正式 session history 提供，前端只在需要时拉取并恢复显示。

## 六、流式渲染策略

### 6.1 普通消息模式

在普通长度消息下，保持当前的实时流式体验。

也就是说：

- token 到来后继续快速反映到 UI
- 不主动引入明显延迟
- 不让正常短消息体验因为长消息问题而退化

### 6.2 长消息稳定刷新模式

当消息增长到设定阈值后，切换到“稳定刷新模式”。

该模式下的核心变化不是停止流式，而是：

- token 先进入内存 buffer
- 前端按固定时间片批量刷新 UI
- UI 刷新频率低于 token 到达频率

这样用户看到的仍然是“内容持续长出来”，但不会因为每个 token 都触发一次长 markdown 重算而卡顿。

### 6.3 设计意图

这个策略的本质是：

**把“token 输入频率”和“UI 渲染频率”解耦。**

也就是说：

- 后端仍然可以高频输出 token
- 前端只以稳定频率消费并更新大消息 UI

### 6.4 本次不强制引入的能力

本方案当前不要求一步到位引入：

- transcript 虚拟列表
- 大规模渲染架构重写
- 复杂消息分页系统

本次优先使用更小的改动面解决实际卡顿。

## 七、前端存储收敛方案

### 7.1 `workspaceStore` 的当前职责

`workspaceStore` 当前已收敛为管理：

- session 元信息
- 当前 session 选择
- 项目展开与搜索等 UI 状态
- 最近 `10` 轮消息缓存或其轻量引用

它不应继续作为完整历史消息的长期容器。

### 7.2 不再保留旧的全量消息持久化路径

在新 UI 下，旧的“把完整 `sessions.items` 持久化到 localStorage”方案不继续保留。

原因很直接：

- 会持续放大序列化成本
- 会持续放大 rehydrate 成本
- 会迫使渲染层长期面对更大的历史数组
- 会让前端不知不觉变成消息真相源

### 7.3 后端拉取历史负责渲染（当前已落地）

历史消息的正确归属应改为：

- 后端保存
- 前端按 session 拉取
- 前端把拉到的 transcript archive 转换成渲染所需 rounds 与 items

前端缓存只是加速层，不再是唯一来源。

### 7.4 本地缓存边界

前端本地缓存仅保留：

- 最近 `10` 轮
- 当前运行中的流式 overlay
- 必要的 UI 元信息

前端不长期缓存：

- 超过 `10` 轮的历史
- 全量 receipt 明细历史
- 完整 execution log

## 八、分步实施策略

### 8.1 第一步：修流式更新频率（已完成）

优先改 `useExecutionOverlay`，把当前 token 级别即时写 UI 的策略改成：

- 普通消息快速刷新
- 长消息按稳定节奏批量刷新

这一步直接作用于当前最明显的卡顿源。

### 8.2 第二步：收缩前端消息持久化边界（已完成）

接着改前端 store，使其不再长期持久化完整历史消息。

目标是让：

- `workspaceStore` 变轻
- localStorage 体积可控
- session 越长，前端长期负担不再线性增长

### 8.3 第三步：接后端历史拉取链路（已完成最小版本）

在前端不再保留完整历史后，由后端承担 transcript history 拉取。

这一步已经建立了新 UI 下唯一保留的消息真相源链路，并支持 `user-message / agent-update / action-receipt / assistant-message` 的完整回放。

### 8.4 第四步：补充 transcript 渲染保护（剩余事项）

如果在完成前面三步后，长会话场景仍有明显渲染压力，再补 transcript 层的额外保护。

例如：

- 更保守的长消息渲染策略
- 更明确的消息窗口控制

但这一步不应先于前面的链路收敛。

## 九、验证标准

本次修复完成后，至少要能验证以下结果：

### 9.1 长消息场景下 UI 更稳定

- assistant 长回复持续输出时不再出现明显卡顿
- 页面滚动和输入响应不应被明显阻塞
- 用户仍能感知到内容在持续流式生成

### 9.2 普通消息体验不退化

- 短消息仍保持接近实时的流式输出
- 不因长消息优化而让普通回复看起来迟钝

### 9.3 前端持久化体积显著收敛

- 本地持久化不再保存完整历史消息
- 最近 `10` 轮缓存边界明确且固定
- session 和历史增长时，本地持久化压力不再持续膨胀

### 9.4 历史完整性由后端保障

- 超过 `10` 轮的历史仍可从后端恢复并展示
- 前端缓存不是唯一来源
- 清空前端缓存不会造成历史不可恢复

### 9.5 新 UI 只有一条消息路径

- 不再存在旧方案与新方案双轨并存
- 渲染层只围绕新的消息来源链路组织

---

## 十、非目标与剩余事项

当前仍未包含以下内容：

1. 完整记忆系统实现
2. Candidate / Durable Memory 设计与落地
3. 完整 Active Workspace 结构设计
4. 大规模 transcript UI 重写
5. 长期保留双轨兼容方案

当前的边界仍需保持清晰：

**先修长消息卡顿与消息存储模型，再继续动记忆体系。**

---

## 十一、结论

这次问题的正确修法不是取消流式，也不是继续给前端堆更多本地历史，而是：

1. 保留 token 流式体验
2. 仅在长消息场景下降低 UI 刷新频率
3. 把完整历史消息真相源收回后端
4. 让前端只保留最近 `10` 轮和当前渲染所需数据
5. 在新 UI 下直接清掉旧的全量消息持久化路径

当前这条链路已经把 ReflexionOS 的 workspace 从“高频流式更新 + 长消息 markdown + 大对象持久化耦合”收敛到了“稳定流式更新 + 后端 transcript 真相源 + 前端轻量缓存”的单线路结构。

这既能解决当前卡顿问题，也能为后续记忆体系建设提供更清晰的前后端边界。
